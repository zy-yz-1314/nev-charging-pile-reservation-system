# 02 · 面试口述版 · 技术实现

> 用途:面试官说"讲一下你这个项目"时的口述脚本 + 高频追问应答。
> 节奏:电梯演讲 30s → 背景 1min → 架构 2min → 四层深讲 5min → 留时间给追问。
> 口语化,**不要照念文档**,讲"为什么这么做"比讲"做了什么"更得分。

---

## 1. 30 秒电梯演讲(开场必背)

> "我做的 PowerQueue 是一个新能源充电桩的**智能调度平台**。和普通充电管理 CRUD 不一样的地方在于,它有一套**四层智能化架构**:第一层是多因子打分的智能匹配加排队队列,第二层是需求预测和动态定价,第三层是 WebSocket 实时推送和充电计划向导,第四层接了 DeepSeek 做自然语言助手——用户说一句'我在国贸,着急,去哪最快充上',系统就能直接给出推荐。整个系统的地基是高并发抢桩的防超卖,我用分布式锁加乐观锁做了三道防线,20 个并发抢一个桩只有一个能成功。"

**这 30 秒要命中四个关键词**:智能调度(不是 CRUD)、四层架构、LLM 杀手锏、防超卖地基。

---

## 2. 项目背景(1 分钟)

> "背景是新能源车主在高峰时段充电有两个痛点:一是站点爆满只能干等,系统被动地让你去翻列表手动找桩;二是排队体验差,前面还有几个人、要等多久完全不知道。
>
> 所以我把项目定位成**调度系统**——调度的本质是**系统主动做决策**,而不是被动执行用户的点击。在这个定位下,我设计了四层智能化架构,从下往上:调度智能、预测智能、体验智能、最上面是 LLM 自然语言交互。"

---

## 3. 架构总览(2 分钟,画图讲)

> "整体是分层依赖的,下层不依赖上层,上层编排下层:
>
> - 最底下是地基:高并发抢桩防超卖,加上 Cache-Aside 缓存和桩状态机。这一层我一开始就做扎实了,因为上面四层所有的读路径都要依赖它。
> - L1 调度智能:智能匹配推荐引擎 + 等待队列。
> - L2 预测智能:需求预测 + 动态定价。
> - L3 体验智能:WebSocket 推送 + 充电计划向导。
> - L4 LLM 助手:编排下面所有接口,输出自然语言。
>
> 设计原则是**上层出问题要能降级到下层**,比如 LLM 挂了,要自动降级到 L1 的智能匹配页面,而不是整个系统瘫掉。"

---

## 4. 四层逐层深讲

### 4.1 地基:高并发抢桩防超卖(必考,先讲透)

> "抢桩的核心矛盾是:黄金时段几十个车主同时抢同一个空闲桩,怎么保证**不超卖**(一个桩不被两个人抢到)、**不重复**(一个用户不重复下单)。
>
> 我用了**三道防线**:
> 1. **Redis 缓存降压**:热门站点状态缓存在 Redis,轮询和列表查询优先读缓存,降低 MySQL 压力;状态变更后主动失效。
> 2. **Redisson 分布式锁**:按 `lock:pile:{pileId}` 加锁,把同一桩的并发请求**串行化**,同时校验用户有没有进行中的订单防重复。
> 3. **DB 乐观锁兜底**:锁内重读后执行
>    `UPDATE charging_pile SET status='RESERVED', version=version+1 WHERE id=? AND status='IDLE' AND version=?`,
>    按**影响行数**判定抢到没有,0 行就是被别人抢了。
>
> 用三道是因为**每一道都有它兜不住的场景**:缓存有一致性窗口不能单独靠它;分布式锁有锁提前释放的风险(锁租约到了但事务还没提交);所以最后必须有乐观锁这个和事务在一起的兜底——只要 version 对不上,更新就失败。我用 `ReservationConcurrencyTest` 跑了 20 并发抢 1 桩,断言只有 1 个成功。"

### 4.2 L1 智能匹配推荐引擎

> "匹配引擎替代了用户手动翻列表。用户只点'我要充电',传地理位置、车型、目标电量,系统算 Top-N。
>
> 打分公式是
> `score = w1×(1/距离) + w2×(1/等待时间) + w3×(1/价格) + w4×功率匹配度`,
> 四个因子加权求和,权重后台可配。
>
> 这里有几个工程细节值得说:
> - **归一化**:距离是 km、时间是 min、价格是元,量纲不同,直接加价格项会被淹没。所以先在候选集里做 min-max 归一化到 [0,1],而且对'越小越好'的项取反(距离越近分越高)。
> - **功率匹配度**:`min(车支持功率, 桩额定功率) / 桩额定功率`。车只支持 60kW 的话,派 180kW 超充就是浪费,匹配度低,这样能引导用户去合适的桩。
> - **地图距离走第三方 API**(高德/百度),结果按起终点缓存 5~10 分钟,避免高频调外部接口。
> - 核心难点其实是**等待时间的预估**,这个我放在队列那块讲。"

### 4.3 L1 等待队列 + 等待时长预估

> "原来的问题是桩满了直接抛 `PILE_NOT_IDLE`,用户体验是'手慢了'就没了。我加了排队机制:桩状态从 4 个扩成 5 个,多了 `QUEUED`。
>
> 队列我用 **Redis Sorted Set** 实现,`queue:pile:{pileId}`,score 是入队时间戳,member 是 userId。为什么用 Sorted Set 而不是继续用分布式锁?因为队列的本质是**有序 + 可排名**,Sorted Set 天然支持:ZCARD 看总人数、ZRANK 看我排第几、ZPOPMIN 取队首,这些分布式锁做不了。
>
> 等待预估:`预估等待 = 我前面的人数 × 单桩平均充电时长`,平均时长取这个桩近 7 天 FINISHED 订单的 AVG(duration)。更准的版本还可以把'当前正在充的桩还剩多久'算进去。
>
> 轮到用户时 WebSocket 推送,给 **10 分钟确认窗口**,超时就 ZPOPMIN 下一位,并给超时用户一个漏充提醒。
>
> 这里有个**关键设计**:队列存 Redis,桩的物理状态(CHARGING/RESERVED)还是乐观锁保护,**两者解耦**。这样排队写入不会干扰抢桩那条 `WHERE status='IDLE' AND version=?` 的更新。"

### 4.4 L2 动态供需定价

> "定价是调度的核心手段。公式是
> `最终电价 = 基础电价 × 时段系数 × 站点负载系数`。
> 时段系数:谷时 0–8 点 0.7、平段 1.0、峰时 17–21 点 1.5;
> 负载系数:空闲率大于 50% 是 0.9 引流,小于 20% 是 1.3 分流。
> 思路是用价格杠杆把人从爆满站挤到空闲站,供需自趋均衡。
>
> 一个容易被追问的点是**防震荡**:如果占用率在 50% 或 20% 边界来回抖,价格会反复跳。所以我加了**滞后区间**,比如 45%~55% 维持原档,只有越过滞后才切换。
>
> 负载系数和三色预警(绿/黄/红)共用同一份占用率数据,这样定价和预测是一致的。"

### 4.5 L2 分时段需求预测

> "预测解决的是'前置预警',别等爆满了才让人排队。
> 维度是'站点 + 小时 + 星期',统计历史占用率,用**移动平均 + 季节性分解**——周一早高峰和周末的逻辑不一样,按星期做季节性分解能捕捉到这种周期性。
> 这里**特意没用深度学习**,一是数据量没那么大,二是时间序列这种周期性用移动平均就能拿到不错的效果,工程上更轻、更好维护。结果提前算好放缓存,定时任务每天凌晨预计算。"

### 4.6 L3 WebSocket 实时推送

> "原来是前端 5 秒轮询拉全量,延迟高还浪费资源。我换成事件驱动:桩状态一变,后端统一广播,前端只**增量刷新**那一个桩。
> 端点是 `/ws/charging`,连接时绑定 userId。
> 推送文案比如:'您关注的 3 号快充桩刚刚空闲,距离您 800m,预计保留 5 分钟'。
> 可靠性上做了心跳保活 + 离线消息补偿:没投递成功的暂存 Redis,下次连上补推,保证不丢。"

### 4.7 L4 LLM 自然语言助手(杀手锏,重点讲)

> "这是项目最区别于普通 CRUD 的部分。用户直接说'我在国贸,着急,去哪最快充上',系统给推荐。
>
> 我用的是**两阶段编排**,核心思想是**让大模型只组织语言、不产生数据**,从架构上杜绝幻觉:
> - **阶段一**:把口语丢给 DeepSeek 做**意图识别 + 槽位提取**,抽出结构化参数{地点、是否着急、目标电量、车型}。
> - **中间**:后端拿这些参数去**串联平台现有的真实接口**——地图 API 算路、匹配推荐、等待预估、动态定价,拿到的全是平台实时数据。
> - **阶段二**:把这些**结构化实时数据 + 用户原始诉求**作为 context 再喂给 DeepSeek,让它**只做自然语言组织**,输出推荐文案 + 可跳转的操作链接。
>
> 容错上,我把 AI 调用封成统一 SDK,配了**超时、熔断、降级**:两阶段都有超时,任一阶段失败或熔断,直接降级返回 L1 智能匹配的结构化列表——没有自然语言文案,但不报错,用户照样能用。"

## 5. 高频面试题 Q&A(核心价值区)

### ▶ 抢桩 / 并发

**Q:分布式锁 + 乐观锁,是不是重复了?有分布式锁不就够了?**
A:不重复,作用层级不同。分布式锁解决**并发串行化**,降低数据库争用;但它有风险——锁租约到了可能在事务提交前就释放(或者 Redis 主从切换瞬间锁丢失)。乐观锁是**和事务在一起**的最后一道防线,只要 version 对不上更新就失败。所以是"锁挡住大多数、乐观锁兜住锁失效的极端情况",纵深防御。

**Q:为什么不用 `SELECT ... FOR UPDATE` 悲观锁?**
A:悲观锁持有时间长、并发吞吐低,而且 MySQL 行锁在 RR 隔离下还可能加间隙锁影响相邻行。乐观锁在冲突不极端时性能更好;真到了冲突极高的场景,我前面还有分布式锁先串行化,把冲突压到很低,乐观锁失败重试概率很小。三者是配合关系。

**Q:锁在事务还没提交就释放了,中间有人抢怎么办?**
A:这就是我强调乐观锁兜底的原因。锁释放后另一个请求能进来,但它在锁内重读到的 version 已经是旧值(或者 status 已不是 IDLE),`WHERE version=?` 命中 0 行,抢失败。所以不会超卖,只是那个用户得到"手慢了"的提示。

### ▶ 智能匹配

**Q:打分公式里 w1~w4 的权重怎么定?**
A:初始值按业务经验给(比如距离权重最高),但我把权重存在 Redis 的 `config:score_weights`,**后台可配**。后续可以按用户点击/转化做 A/B 或者简单的线性回归去校准——哪个因子调高之后用户采纳率上升,就往那个方向调。MVP 阶段先经验值 + 可配置。

**Q:距离、时间、价格量纲不一样,直接加合理吗?**
A:不合理,这正是工程上要处理的。我在候选集内做 **min-max 归一化**到 [0,1],对"越小越好"的项(距离、等待、价格)做 `f_norm = (max − f)/(max − min)` 反转,功率匹配度本身就是 [0,1]。归一化后再加权,各因子才可比。

**Q:等待时间预估准不准?**
A:均值法是 baseline,精度有限。提升方向:把"当前正在充电桩的剩余时间"算进去——用 `startTime + 平均充电时长 − now` 估算每个进行中订单的剩余,累加比单纯乘均值准。再往后可以上指数加权(近期订单权重更高)。但 MVP 用均值已经能给出可解释的估计,关键是**告知用户这是预估值**,管理预期。

### ▶ 等待队列

**Q:为什么用 Redis Sorted Set,而不是用消息队列或者继续用分布式锁?**
A:队列的核心需求是**保序 + 可查排名 + 可按优先级插队**。分布式锁只能互斥,做不了排名;专业 MQ(Kafka/RabbitMQ)偏重吞吐和异步解耦,但我这里需要的是**实时查"我排第几"**,Sorted Set 的 ZRANK/ZCARD 天然支持,而且 score 还能叠加会员优先级。轻量、够用、契合场景,不必引入 MQ。

**Q:你加了 QUEUED 状态,会不会破坏原来的乐观锁 status 流转?**
A:不会,这是设计关键。`QUEUED` 是**对外展示/业务语义**(这桩满载但可排队),真正的队列数据存 Redis Sorted Set,**不写 charging_pile 表**。桩在库里的物理状态仍是 IDLE/RESERVED/CHARGING,继续被乐观锁的 `WHERE status='IDLE' AND version=?` 保护。两者解耦,排队不污染抢桩。

**Q:10 分钟确认窗口,用户没确认怎么办?**
A:定时任务(或 Redis Key 过期事件)扫超时队列,对该用户 ZPOPMIN 移出,推一条漏充提醒;同时 ZPOPMIN 取下一位、给他推"轮到你了"。给下一位也要重新计 10 分钟窗口。

### ▶ 动态定价

**Q:动态定价会不会让价格反复震荡?**
A:会,这是边界问题。我的处理是**滞后区间(hysteresis)**:在 50%、20% 阈值附近留缓冲带(比如 45%~55% 维持原档),只有真正越过滞后区才切换档位,避免占用率抖动导致价格跳。

**Q:负载系数的占用率怎么实时拿?**
A:和空闲桩统计共用——站点总桩数固定,空闲桩数走缓存(抢桩变更时失效),空闲率实时可算。预测用的也是这份数据,保证定价和预警一致。

### ▶ WebSocket

**Q:怎么保证消息不丢?**
A:心跳保活检测连接;投递时如果用户不在线,消息暂存 Redis(按 userId 队列),下次连接先补推。强一致场景可以加 ACK——客户端收到回执,服务端没收到 ACK 就重发。MVP 做到"暂存 + 补推"已经够用。

**Q:多实例部署时,用户连在 A 实例,事件发在 B 实例怎么办?**
A:本地 `userId → Session` 只在本实例有效,多实例要上 **Redis Pub/Sub**(或 MQ):状态变更发布到频道,所有实例订阅,各自推自己持有的本地连接。这是从单机到集群的自然演进路径。

### ▶ LLM 助手

**Q:大模型幻觉怎么防?它瞎编一个价格怎么办?**
A:这是我最在意的问题,所以做成两阶段。**模型绝不直接产出价格/桩位数字**——阶段一它只做意图和槽位提取(抽参数);真正的距离、价格、等待时长都是后端在阶段中间**调平台真实接口**拿的;阶段二把这些真实数据作为 context 给模型,它**只负责组织语言**。数字源头在后端,模型没有"编"的机会。再加业务 Prompt 约束和结果校验(回填的桩 id 必须存在于推荐结果集)。

**Q:为什么两阶段,不直接一次 Function Calling 搞定?**
A:一阶段 Function Calling 也能做,但两阶段的好处是**编排控制权在后端**:中间这步我可以串多个接口、加缓存、加校验,逻辑是确定性的、可测试的、不依赖模型每轮都正确调工具。两阶段把"理解用户"和"组织输出"这两个 LLM 擅长的活儿,和"查实时数据"这个必须确定性的活儿,**清晰分开**了。

**Q:DeepSeek 接口超时或挂了怎么办?**
A:统一 SDK 封装,两阶段各自超时(比如各 3s);用 Resilience4j 配熔断器,错误率到阈值就熔断;**降级策略是直接返回 L1 智能匹配的结构化推荐列表**——没有自然语言文案,但用户照样能选桩下单。LLM 是锦上添花,不能成为单点故障拖垮核心链路。

**Q:接 LLM 这块你怎么测?**
A:槽位提取阶段可以用固定样例做断言(输入"我在国贸着急" → 期望抽出 location、urgency);阶段二的语言输出不好做精确断言,我主要断言**回填的数据字段**(桩 id、价格)确实来自推荐结果集,即校验"数据真实性"而非"文案措辞"。

### ▶ 项目反思

**Q:这个项目你觉得还有什么不足 / 可以优化的?**
A:几个方向:
1. 等待预估还比较粗,可以上指数加权或把进行中订单的剩余时间算进去。
2. 定价系数目前是规则配置,数据攒够后可以尝试用历史数据拟合更优系数。
3. WebSocket 多实例还没上 Pub/Sub,目前是单机 Session 映射,集群部署要补。
4. LLM 两阶段调用有两次 RTT,延迟偏高,可以考虑流式输出改善体验。
能讲出不足反而说明你真做过、真想过——比"完美无缺"可信得多。

## 6. 口述节奏与禁忌

- **节奏**:总长控制在 8~10 分钟可缩可放,看面试官兴趣。他对哪层感兴趣就展开哪层,别每层都平均用力。
- **禁忌**:
  - 别背名词堆术语(说了"熔断"就准备好讲 Resilience4j 的滑动窗口、半开状态)。
  - 别编没做过的压测数字(没压过就说"理论分析 + 单元测试验证",不要瞎报 QPS)。
  - 被追到不会的,诚实说"这块我目前是 XX 方案,你说的方向我没深入,但我理解可以这样改进……"——比硬编强。
