# 03 · 数据库 / 缓存 / 接口设计

> 用途:落地实现的工程蓝图。所有 DDL 都是**在现有 `sql/powerqueue.sql` 基础上的增量**,不破坏已有表结构与并发模型。
> 原则:`status` 字段已是 `VARCHAR`,状态扩展**不改表结构**只扩枚举语义;队列走 Redis 不落库,避免污染乐观锁。

---

## 一、数据库设计

### 1.1 现有表盘点(不动)

| 表 | 作用 | 关键字段 |
|----|------|----------|
| `user` | 用户 | role、balance、car_plate |
| `station` | 充电站 | lng/lat、status |
| `charging_pile` | 充电桩 | type(FAST/SLOW)、power、price、status、**version(乐观锁)** |
| `reservation` | 订单 | order_no、pile_id、duration、power_used、amount、status |

### 1.2 状态扩展(不改结构,只扩语义)

**`charging_pile.status`**:`VARCHAR` 已支持,枚举从 4 扩到 5:

```
[IDLE, RESERVED, CHARGING, FAULT]  →  [IDLE, RESERVED, CHARGING, FAULT, QUEUED]
```

- `QUEUED`:桩满载(CHARGING/RESERVED)且**存在等待队列**,对外提示"可排队"。
- 物理上桩的库内状态仍由乐观锁保护;`QUEUED` 由服务层在"桩繁忙 + Redis 队列非空"时对外推导,**不直接写入抢桩的 `WHERE status='IDLE'` 链路**。

**`reservation.status`**:新增 `QUEUED`(已排队待确认)与 `WAITING_CONFIRM`(已轮到、待用户确认)两个业务态,枚举变为:

```
[PENDING, CHARGING, FINISHED, CANCELLED]  →  [PENDING, QUEUED, WAITING_CONFIRM, CHARGING, FINISHED, CANCELLED]
```

> 说明:若希望最小侵入,`QUEUED/WAITING_CONFIRM` 可只在 Redis 体现,订单落库仍用 `PENDING` + 一个 `queue_state` 字段区分。下文 DDL 采用**显式枚举 + 快照字段**方案,便于查询与审计。

## 二、Redis 缓存设计

### 2.1 现有 Key(保留)

| Key | 类型 | TTL | 用途 |
|-----|------|-----|------|
| `station:piles:{stationId}` | String(JSON) | 30s | 站点充电桩列表缓存(Cache-Aside) |
| `order:seq` | String(counter) | 永久 | 订单号自增序列 |
| `lock:pile:{pileId}` | Redisson Lock | 租约 5s | 抢桩分布式锁 |

### 2.2 新增 Key 表

| Key | 类型 | TTL | 用途 | 归属层 |
|-----|------|-----|------|--------|
| `queue:pile:{pileId}` | **Sorted Set** | 按队空清理 | 等待队列:score=入队时间戳(可叠加会员优先级),member=userId | L1 |
| `queue:confirm:{userId}` | String | **600s** | 10 分钟确认占位窗口标记;过期触发下一位轮换 | L1 |
| `pile:avg_duration:{pileId}` | String | 1h | 单桩近 7 天平均充电时长(分钟),预估等待用 | L1 |
| `station:load:{stationId}` | Hash | 10s | 实时空闲率:`{idle, total, idleRate, loadLevel}` | L2 |
| `forecast:station:{stationId}:{dow}:{hour}` | Hash | 25h | 预测占用率 + 三色档(凌晨预计算) | L2 |
| `config:score_weights:{profile}` | Hash | 5m | 打分权重缓存(DB 为源) | L1 |
| `config:pricing` | Hash | 5m | 时段/负载系数缓存(DB 为源) | L2 |
| `distance:{lng4}:{lat4}:{stationId}` | String | 600s | 地图算路结果缓存(精度截断到 4 位防 Key 爆炸) | L1/L4 |
| `ws:offline:{userId}` | **List** | 7d | 离线消息补偿队列,RPUSH 暂存 / 连上 LPOP 补推 | L3 |
| `ws:online:{userId}` | String(instanceId) | 心跳 | 用户当前连在哪个实例(多实例路由用) | L3 |
| `ai:rate:{userId}` | String(counter) | 60s | LLM 助手简单限流(如 10 次/分钟) | L4 |
| `ai:circuit` | Resilience4j 状态 | — | DeepSeek 熔断器共享状态(集群可走 Redis) | L4 |

### 2.3 Sorted Set 队列操作对照

| 业务动作 | Redis 命令 |
|----------|-----------|
| 用户入队 | `ZADD queue:pile:{pileId} {score} {userId}` |
| 我排第几(0 起) | `ZRANK queue:pile:{pileId} {userId}` |
| 队伍总人数 | `ZCARD queue:pile:{pileId}` |
| 我前面还有几人 | `ZCARD - ZRANK - 1` |
| 取队首并移除 | `ZPOPMIN queue:pile:{pileId}` |
| 我退出排队 | `ZREM queue:pile:{pileId} {userId}` |
| 看 ahead N 名 | `ZRANGE queue:pile:{pileId} 0 N-1 WITHSCORES` |

> **优先级设计**:score 默认 = 入队时间戳;若要做会员优先级,`score = baseTimestamp - memberLevel × 10^6`(等级越高 score 越小越靠前),保持单调可排序。

### 2.4 缓存一致性策略

- **Cache-Aside 不变**:读未命中查 DB 回填;写(状态变更)后主动 `evict`。
- **队列与缓存解耦**:队列直接操作 Sorted Set,不经过 `station:piles` 缓存,避免互相失效放大。
- **配置类缓存**:后台改权重/系数时,删 `config:*` 对应 Key,下次请求回源 DB 并回填(旁路一致性)。

### 3.4 LLM 助手完整调用链路(细化)

```
POST /api/ai/chat  { query, lng, lat }
        │
        ▼  ① 限流:ai:rate:{userId} + 熔断检查(Resilience4j)
        │
        ▼  ② DeepSeek 阶段一(意图/槽位提取,超时 3s)
        │     Prompt: 业务约束系统提示 + 用户 query + {lng,lat}
        │     强制 JSON Schema 输出: { location, urgency, targetEnergy, carPowerKW }
        │
        ▼  ③ 后端确定性编排(全部平台真实数据):
        │     ├─ location → 地图 API → 候选站点 + distance 缓存
        │     ├─ /api/recommend     → 排序桩(L1)
        │     ├─ /api/queues/.../estimate → 等待时长(L1)
        │     └─ /api/price/calc    → 动态价(L2)
        │
        ▼  ④ DeepSeek 阶段二(语言组织,超时 3s)
        │     Prompt: 「仅依据下方真实数据组织推荐,禁止编造数字」+ slots + 真实数据 JSON + 用户 query
        │     输出: 自然语言文案 + 推荐桩 id 列表
        │
        ▼  ⑤ 结果校验:回填的 pileId ⊆ ③ 候选集(防模型塞不存在的桩)
        │
        ▼  ⑥ 落库 ai_chat_log + 返回 { reply, piles[], degraded:false }
        │
   任一阶段超时/熔断 → 降级:直接返回 ③ 的结构化推荐列表(degraded:true,无文案)
```

## 四、状态机与事件流

### 4.1 桩状态流转(含 QUEUED 语义)

```
            抢桩成功
   IDLE ───────────────► RESERVED ──开始充电──► CHARGING
    ▲ ▲                    │                       │
    │ │ 取消/超时           │ 轮到确认              │ 结束/取消
    │ │                    ▼                       │
    │ │              WAITING_CONFIRM               │
    │ │             (队列首位+10min窗口)            │
    │ │                    │                       │
    │ │     入队(满桩)    │                       │
    │ └────────── QUEUED ◄─┘                       │
    │              │                               │
    │              │ 用户在队期间,对外推导显示      │
    │              │ "QUEUED"(可排队)             │
    │              ▼                               │
    │           (桩物理仍 CHARGING,                │
    │            队列存 Redis,不影响乐观锁)         │
    └──────────────────────────────────────────────┘
                          故障 → FAULT
```

> 关键:虚线 `QUEUED` 是**对外推导态**,不写库抢桩链路;`RESERVED/CHARGING/IDLE` 的物理流转仍由 `ChargingPileMapper.grabPile/updateStatus` + 乐观锁保护。

### 4.2 广播订阅模型

- **生产端**:`grabPile/startCharging/finish/cancel`、队首轮换、定价档变 —— 统一 `applicationContext.publishEvent(new ChargingEvent(...))`。
- **消费端**:`ChargingEventBroadcaster` 监听事件 → 查 `ws:online:{userId}`(或本地 Session 表)→ 推给订阅了该 `stationId/pileId` 的连接。
- **多实例**:事件同时 `PUBLISH charging:events` 到 Redis Pub/Sub,其他实例订阅后推本地连接(单机→集群平滑演进)。
