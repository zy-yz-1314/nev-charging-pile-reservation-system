# PowerQueue · 四层智能化充电调度平台 · 总体设计

> 本文件是整个智能化升级的**唯一需求源(Single Source of Truth)**。
> 任何文档、代码、简历、口述稿出现冲突时,以本文件的业务规则、计算公式、技术选型、优先级表为准。
>
> 适用对象:Java 后端开发(秋招 / 社招)、面试复盘、项目迭代排期。

---

## 0. 角色与定位

- **角色**:资深 Java 后端开发工程师。
- **项目定位**:把一个"被动执行操作"的充电桩 CRUD 平台,升级为**具备自主决策能力的智能调度系统**。
- **核心判据**:能不能在用户只给一句"我要充电"的前提下,由系统**主动算出最优桩、预估等待、动态定价、并兜底异常**——这是"调度系统"与"管理系统"的本质区别。

## 1. 现有系统盘点(v1.0 地基)

升级不是推翻重来,而是在已有的**高并发地基**上往上盖楼。先盘清现有资产:

| 模块 | 现有实现 | 关键代码位置 |
|------|----------|--------------|
| 抢桩防超卖 | 三道防线:Redis 缓存降压 + Redisson 分布式锁 + DB 乐观锁(`version`) | `ReservationServiceImpl#grabPile`、`ChargingPileMapper.xml#grabPile` |
| 缓存策略 | Cache-Aside,`station:piles:{stationId}`,TTL 30s,变更后主动失效 | `ChargingPileServiceImpl` |
| 桩状态机 | `[IDLE, RESERVED, CHARGING, FAULT]` | `PileStatus` 枚举 |
| 满桩处理 | 直接抛 `PILE_NOT_IDLE(2002)`,**无排队机制** | `ResultCode`、`grabPile` |
| 实时刷新 | 前端 **5 秒轮询** `/api/stations/{id}/piles` | `frontend/.../StationDetail.vue` |
| 订单号生成 | Redis 自增序列 `order:seq` | `ReservationServiceImpl#generateOrderNo` |
| 订单结算 | `金额 = 功率 ×(时长/60)× 单价`,**固定单价** | `ReservationServiceImpl#finish` |
| 鉴权 | JWT(jjwt)+ BCrypt,拦截器注入 `UserContext` | `JwtInterceptor`、`JwtUtil` |
| 并发验证 | `ReservationConcurrencyTest`:20 并发抢 1 桩,断言仅 1 成功 | `src/test/...` |

**结论**:并发安全这块地基是稳的(分布式锁 + 乐观锁兜底,锁提前释放也不超卖)。四层智能化的所有读路径(匹配、预测、定价、推送、LLM 编排)都构建在这套状态机与缓存之上,**不破坏乐观锁的 status 流转**。

## 2. 四层智能架构总览

```
┌─────────────────────────────────────────────────────────────┐
│  L4 体验交互层   DeepSeek 自然语言助手(T3 杀手锏)              │
│      口语化诉求 → 意图/槽位提取 → 编排底层接口 → 自然语言推荐     │
├─────────────────────────────────────────────────────────────┤
│  L3 体验智能层   充电计划向导 + WebSocket 全量实时推送            │
│      事件驱动:桩状态变更 → 后端广播 → 前端增量局部刷新            │
├─────────────────────────────────────────────────────────────┤
│  L2 预测智能层   分时段需求预测(移动平均+季节性) + 动态供需定价   │
│      价格杠杆:基础电价 × 时段系数 × 站点负载系数                │
├─────────────────────────────────────────────────────────────┤
│  L1 调度智能层   智能匹配推荐引擎 + 等待队列 + 等待时长预估      │
│      多因子加权打分 + Redis Sorted Set 优先级排队              │
├─────────────────────────────────────────────────────────────┤
│  地基  高并发抢桩(分布式锁+乐观锁) + Cache-Aside + 状态机      │
└─────────────────────────────────────────────────────────────┘
```

设计原则:**下层不依赖上层,上层编排下层**。L4 出问题要能降级到 L1,而不是让整个系统瘫掉。

## 3. 第一层:调度智能(核心基础,P0)

> 调度系统必备能力。不做这层,后面三层都是空中楼阁。

### 3.1 智能匹配推荐引擎(替代手动翻列表)

| 项 | 内容 |
|----|------|
| **用户操作** | APP 仅点击「我要充电」 |
| **入参** | 用户实时地理位置、车型信息(电池容量、支持充电功率)、目标充电电量 |
| **输出** | 综合得分 Top-N 充电桩 |
| **打分公式** | `score = w1 × (1 / 距离) + w2 × (1 / 等待时间) + w3 × (1 / 价格) + w4 × 功率匹配度` |
| **实现逻辑** | 多因子加权排序,**权重 w1~w4 后台可配置**;接入高德/百度地图 API 计算真实行驶距离 |
| **核心难点** | **精准预估排队等待时长**(见 3.2) |

**工程实现要点(防面试追问)**:
- 四个因子量纲不同(距离 km、时间 min、价格 元),必须先**归一化到 [0,1]** 再加权,否则价格项会因数值小被淹没。归一化采用候选集 min-max:`f_norm = (f_max − f) / (f_max − f_min)`(越小越好的项取反)。
- **功率匹配度**定义:`min(车辆支持功率, 桩额定功率) / 桩额定功率`——若车只支持 60kW,派到 180kW 超充也是浪费,匹配度低。
- 地图距离走第三方 API,**结果按 (起点,终点) 维度缓存 5~10 分钟**,避免高频打外部接口(也省钱)。
- 权重 `config:score_weights` 存 Redis,运营在后台改完即时生效,不下发代码。

### 3.2 充电桩等待队列 + 等待时长预估

| 项 | 内容 |
|----|------|
| **桩状态扩展** | `[IDLE, RESERVED, CHARGING, FAULT]` → **`[IDLE, RESERVED, CHARGING, FAULT, QUEUED]`** |
| **原缺陷** | 桩满直接抛 `PILE_NOT_IDLE`,无排队机制 |
| **业务逻辑** | 无空闲桩时,用户**自动进入对应桩等待队列**;`预估等待时间 = 排队人数 × 单桩平均充电时长`;轮到用户时**推送消息**,设置 **10 分钟确认占位窗口** |
| **技术实现** | **Redis Sorted Set** 实现优先级排队队列,搭配 **WebSocket** 实时推送;**淘汰原始 5 秒轮询**方案 |

**状态机衔接(关键设计,不破坏乐观锁)**:
- `QUEUED` 是对外的**业务/展示状态**,表示"此桩满载且已有人排队,可继续排队"。
- 队列实体不落 `charging_pile` 表,而存 Redis Sorted Set `queue:pile:{pileId}`,`score = 入队时间戳(可叠加会员优先级权重)`,`member = userId`。
- 桩的物理状态(`CHARGING/RESERVED`)仍由乐观锁 `version` 保护,**队列与 DB status 解耦**,避免排队写入干扰抢桩的 `UPDATE ... WHERE status='IDLE' AND version=?`。
- 取队首:`ZPOPMIN`;查我排第几:`ZRANK`;查前面还有几人:`ZCARD - ZRANK - 1`。

**等待时长预估公式细化**:
```
预估等待 = (我前面的人数) × (单桩平均充电时长)
单桩平均充电时长 = AVG(reservation.duration) WHERE pile_id=? AND status='FINISHED'   -- 取近 7 天
```
> 增强版:再把"当前正在充电桩的剩余时间"算进去(用 `startTime + avgDuration − now`),精度更高。MVP 先用均值。

**10 分钟确认窗口**:轮到用户时 WebSocket 推送 → 用户 10 分钟内点确认即抢桩,超时自动 `ZPOPMIN` 下一位、给超时用户一次漏充提醒。

## 4. 第二层:预测智能(系统预判供需,P1/P2)

### 4.1 分时段站点需求预测(P2)

| 项 | 内容 |
|----|------|
| **解决痛点** | 高峰期站点爆满后才允许排队,**无前置预警** |
| **数据依据** | 历史订单,维度为「**充电站 + 小时时段 + 星期**」,统计历史占用率 |
| **算法选型** | **轻量化时间序列(移动平均 + 季节性分解)**,**无需深度学习 / 大模型** |
| **功能** | 提前 N 小时标记站点负荷,前端三色展示:**🟢宽松 / 🟡较忙 / 🔴高峰** |

**实现要点**:
- 统计维度 `station_id × day_of_week × hour`,占用率 = `占用桩数 / 总桩数`,按历史若干周做移动平均,叠加按星期的季节性权重。
- 结果写缓存 `forecast:station:{stationId}:{dayOfWeek}:{hour}`,定时任务每天凌晨预计算一次,避免实时算。
- 三色阈值后台可配(如占用率 <50% 绿、50%~80% 黄、>80% 红),与 4.2 的负载系数**共用同一份占用率数据**。

### 4.2 动态供需定价(调度核心手段,P1)

**最终电价计算公式**:

```
最终电价 = 基础电价 × 时段系数 × 站点负载系数
```

**时段系数**:

| 时段 | 时间 | 系数 |
|------|------|------|
| 谷时 | 0:00–8:00 | **0.7** |
| 平段 | 其余 | **1.0** |
| 峰时 | 17:00–21:00 | **1.5** |

**站点负载系数**:

| 站点空闲率 | 系数 | 意图 |
|-----------|------|------|
| **空闲率 > 50%** | **0.9** | **引流**(降价吸客) |
| 50% ≤ 空闲率 ≤ 20%(中间区) | **1.0** | 平衡 |
| **空闲率 < 20%** | **1.3** | **分流控流**(涨价疏导) |

> 依靠**价格杠杆**自动平衡充电供需:空闲站降价把人吸过去,爆满站涨价把人挤走,负载自趋均衡。

**防震荡设计(加分细节)**:在 50% / 20% 边界引入**滞后区间(hysteresis)**,如 45%~55% 维持原档不变,避免占用率在阈值附近抖动导致价格反复跳。

## 5. 第三层:体验智能(个性化用户交互,P1/P2)

### 5.1 充电计划智能向导(P2)

| 项 | 内容 |
|----|------|
| **用户输入示例** | "每周一三五通勤,单程 30km,无家用充电桩" |
| **系统输出** | 固定充电站推荐、**周期性自动预约充电**、充电漏充消息提醒 |
| **技术方案** | **定时任务 + 用户画像 + 业务规则引擎** |

**实现要点**:
- 用户画像字段:通勤日、单程里程、是否有家充、车型耗电(kWh/100km)。
- 规则引擎:按耗电推算"何时电量会到警戒线",反推最晚充电时间,匹配固定站点,生成周期预约计划。
- 定时任务(Spring `@Scheduled` / XXL-JOB)到点自动发起预约;漏充(用户没在预约时间到)触发消息提醒。

### 5.2 WebSocket 全量实时推送(P1,替换 5 秒轮询)

| 项 | 内容 |
|----|------|
| **旧方案弊端** | 定时拉取全量数据,**延迟高、资源消耗大** |
| **事件驱动链路** | 充电桩状态变更 → 后端统一广播 → 前端**增量局部刷新** |
| **推送文案示例** | `您关注的 3 号快充桩刚刚空闲,距离您 800m,预计保留 5 分钟` |

**实现要点**:
- 端点 `/ws/charging`,连接建立时绑定 `userId`,服务端维护 `userId → Session` 映射(本地 `ConcurrentHashMap`,多实例再上 Redis pub/sub)。
- 状态变更点(`grabPile/startCharging/finish/cancel`、队首轮换)统一发 ` ChargingEvent`,事件总线广播给关注该站/该桩的用户,**只推变更的那一个桩**(增量),不是全量重拉。
- 心跳保活 + 离线消息补偿(未投递的暂存 Redis,下次连上补推),保证消息不丢。

## 6. 第四层:T3 杀手锏 · LLM 自然语言交互助手(P1)

> **项目差异化亮点、面试记忆点。** 体现大模型工程落地、多接口编排、服务容错设计,区别于普通业务 CRUD。

### 6.1 功能描述

- 接入 **DeepSeek 大模型**,搭建自然语言对话交互层。
- **用户交互形式**:直接输入口语化充电需求。
- **示例**:`我人在国贸,着急,现在去哪最快充上`

### 6.2 完整调用链路(两阶段编排,从架构上杜绝幻觉)

```
① 用户口语文本
        │
        ▼
② [DeepSeek · 阶段一] 意图识别 + 槽位提取
   → 结构化参数 { location=国贸, urgency=高, targetEnergy, carModel }
        │
        ▼
③ 后端按参数【串联平台现有实时接口】(全部是平台真实数据,非模型编造):
   ├─ 地图 API:location → 候选站点 + 真实行驶距离
   ├─ /api/recommend     → 综合得分排序桩(L1)
   ├─ 等待时长预估        → 队列等待时间(L1)
   └─ 动态定价计算        → 实时电价(L2)
        │
        ▼
④ 把上述【结构化实时数据】+ 用户原始诉求 作为 context
   [DeepSeek · 阶段二] 仅做"自然语言组织输出",输出:
   → 通顺推荐文案 + 可直接跳转充电的操作链接
        │
        ▼
⑤ 返回前端:自然语言卡片 + 一键跳转按钮
```

**关键约束**:大模型**只组织语言、不产生数据**。所有数字(距离、价格、等待时长)都来自步骤③的平台实时接口,从架构上杜绝模型"编造价格/桩位"。

### 6.3 约束与容错

- 添加**专属业务 Prompt 约束**,杜绝大模型幻觉,**所有推荐数据必须来源于平台实时充电桩数据**。
- **封装 AI 调用统一 SDK**,配置**超时、熔断、降级**策略;大模型服务异常自动**切回原有智能匹配页面**,保障系统稳定。
- 两阶段都加超时(如各 3s),任一阶段失败 / 熔断 → 直接降级返回 L1 智能匹配的结构化列表(无自然语言文案,但不报错)。

### 6.4 项目加分价值

体现 **大模型工程落地、多接口编排、服务容错设计**,区别普通业务 CRUD 项目,面试官印象深刻。

## 7. 技术栈增量(在现有 pom.xml 基础上)

| 用途 | 现状 | 增量 |
|------|------|------|
| 实时推送 | 5 秒轮询 | `spring-boot-starter-websocket` |
| 大模型调用 | 无 | DeepSeek HTTP API(OkHttp / Spring `RestTemplate` / WebClient) |
| 熔断降级 | 无 | Resilience4j(轻量,契合"不堆虚高技术") |
| 定时任务 | 无 | Spring `@Scheduled`(向导 / 预测预计算);规模大了再上 XXL-JOB |
| 地图距离 | 无 | 高德/百度 Web 服务 API(HTTP) |
| 队列 | Redisson 锁 | 复用现有 Redis,**Sorted Set**(无需新中间件) |
| 时间序列 | 无 | 自实现 移动平均 + 季节性分解(无 ML 框架) |

> 选型原则(用户硬约束):**技术方案务实可落地,不使用虚高技术**。所以预测用移动平均而非深度学习,熔断用 Resilience4j 而非全套 Spring Cloud,定时任务用 `@Scheduled` 而非一上来就 XXL-JOB。
